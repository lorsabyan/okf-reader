## Problem

### A. Resource/citation URLs break mid-word

Verified in a live browser at both desktop and mobile widths: long resource URLs wrap using
`break-all`, which breaks between ANY two characters once a line is full — including mid-word,
producing ugly breaks like "ecommerce" splitting into "ecommerc" / "e" across lines, or
"bigquery" splitting into "bigq" / "uery". `break-all` is the wrong tool here: these URLs already
have natural break points (`/`, `_`, `.`) that a less aggressive wrap strategy would prefer.

Current code, verbatim:

    // src/app/(reader)/c/[...slug]/page.tsx:89-105 — current
    {concept.resource && (
      <p className="mt-2 break-all text-sm">
        {isSafeResourceUrl(concept.resource) ? (
          <a
            href={concept.resource}
            target="_blank"
            rel="noreferrer"
            className="inline-flex items-center gap-1 text-primary hover:underline"
          >
            <ExternalLink className="size-3.5 shrink-0" />
            {concept.resource}
          </a>
        ) : (
          concept.resource
        )}
      </p>
    )}

### B. Dev-mode search message is developer-facing copy in an end-user surface

`src/components/SearchDialog.tsx` shows this message whenever Pagefind's index isn't available
(always true under `bun run dev`, since the index is generated by `bun run build`). The README
documents `OKF_BUNDLE=/path/to/bundle bun run dev` as the normal, expected way for a bundle author
to preview their catalog before publishing — that author may not be a JS/Bun-literate developer
(e.g. a data engineer or technical writer maintaining an OKF catalog), so a raw "run bun run build"
instruction reads like a debug/error log line rather than a considered product message.

Current code, verbatim:

## Target

### A. Swap `break-all` for `break-words`

    // target — only the className changes, one word
    <p className="mt-2 break-words text-sm">

`break-words` maps to `overflow-wrap: break-word`, which only breaks inside a word as a last
resort (when a single word is wider than its container) — for URLs with `/`/`_` separators, the
browser will now prefer breaking at those natural points instead of mid-word.

### B. Reword the message

    // target — exact string
    unavailableMessage="Full-text search isn't available in this preview — it's generated by `bun run build`."

(Keep the backtick-quoted `bun run build` — matches this codebase's existing convention of
backtick-quoting commands/identifiers in user-facing strings, e.g. the README and other
`unavailableMessage`-adjacent copy.)

## Review addendum (Fable, post-execution)

The executor correctly implemented target A exactly as written, then caught and precisely
diagnosed a real gap in the plan's own "Done when" criterion: swapping to `break-words` alone
doesn't prevent the anchor from overflowing its container at 375px, because the anchor is
`inline-flex` — its raw text child is an anonymous flex item, and `overflow-wrap` is excluded from
a flex item's min-content size by spec, so the box refused to shrink even though the text wrap
mode was now correct. The executor stayed in scope (flagged it rather than expanding the fix) per
the plan's boundary against further refactoring.

Fixed during review: wrapped `{concept.resource}` in `<span className="min-w-0">…</span>` inside
the anchor (a flex item's specified `display` blockifies, so `min-width` takes effect there) and
added `max-w-full` to the anchor itself (a hard constraint, since the anchor is not itself a flex
item of anything and so isn't shrunk by the min-width mechanism alone). Verified via
`getBoundingClientRect()` at 375px: the anchor now measures exactly its container's width with no
document-level horizontal overflow attributable to it. (A separate, pre-existing, out-of-scope
overflow source remains on some pages — the Shiki code block's intentional `overflow-x: auto`
scroll area, and a plain-HTML "Citations" link rendered by the markdown pipeline with no Tailwind
classes at all — neither was touched by this plan and neither regressed.)
- **Done when**: both checks pass and the diff is limited to the two described one-line changes.
